# 测试与答辩讲解

## 一、为什么要把测试和答辩放在一起讲

对毕业设计来说，测试和答辩其实都在回答同一个问题：你怎么证明这个项目是真的做成了，而不是只把页面和代码堆在一起。测试负责给出工程层面的证据，答辩负责把这些证据讲清楚。

因此，这份文档把两者放在一起，不是为了省事，而是为了让“验证”和“表达”形成闭环。

## 二、本项目的测试重点是什么

这个项目的测试不是追求全面覆盖所有细节，而是优先卡住那些最容易在答辩现场翻车的点。当前更值得重点说明的测试包括：

- `util/tests_configread.py`
  验证配置读取与字符集兼容逻辑。
- `util/tests_startup_scripts.py`
  验证本地启动脚本和样例环境变量是否齐全。
- `templates/front/admin/tests/security-cleanup.test.mjs`
  验证登录态和会话清理逻辑，减少旧标签页残留导致的异常。
- `templates/front/admin/tests/forecast-state.test.mjs`
  验证预测工作台相关状态行为。
- `templates/front/admin/tests/forecast-workspace-config.test.mjs`
  验证预测页配置收口逻辑。
- `main/tests.py`
  验证后端业务接口、车型知识采集、训练命令和部分实验链路。

从答辩角度看，这些测试覆盖已经能说明项目并不是“只凭人工点几下感觉正常”。

## 三、答辩时该如何介绍测试

不要只说“我写了测试用例”，而应该明确说清楚测试类型：

- 有配置与部署层测试，保证换电脑后的环境一致性。
- 有前端状态测试，保证登录态清理和工作台配置不出错。
- 有后端接口和命令测试，保证业务主线和训练命令可运行。

这样老师能迅速感受到你对项目稳定性的关注，而不是把测试当成形式上的补充。

## 四、推荐的答辩演示顺序

最稳的一条演示路径通常是：

1. 先进入首页，展示系统整体定位。
2. 再进入行车日志列表页，展示基础业务能力。
3. 然后进入预测工作台，展示智能分析能力。
4. 接着进入 NASA 实验页，展示研究深度。
5. 最后回到架构图和代码讲解，说明系统实现闭环。

这条顺序的好处是由浅入深，先让老师看到“系统确实能用”，再让老师看到“项目不止能用，还有研究性”。

## 五、老师最常问的几个问题

### 1. 为什么选这个题目

回答重点应放在三点：

- 新能源汽车背景下，行车日志和电池状态分析具有现实意义。
- 单纯做后台管理太浅，所以加入了预测与实验链路。
- 这个题目同时具备工程实现价值和研究扩展空间。

### 2. 项目最大的亮点是什么

建议从三个角度回答：

- 有完整前后端业务闭环。
- 有机器学习主预测链路和深度学习对比实验。
- 有 NASA 公开数据集 SOH / RUL 实验作为研究补强。

### 3. 为什么系统主展示采用机器学习，不直接用深度学习

标准回答思路是：

- 业务数据是结构化表格，更适合稳定、可解释的机器学习方法。
- 深度学习在本项目中承担对比实验和方法探索角色。
- 系统设计优先保证业务稳定输出，再补研究实验深度。

### 4. NASA 实验和系统里的寿命预测是不是一回事

必须明确回答不是一回事，并说明：

- 业务预测面向系统展示和辅助分析。
- NASA 实验面向真实电池退化研究。
- 两者数据来源、标签定义和目标都不同。

## 六、回答问题时最稳的表达方式

建议遵循三个原则：

- 先讲结论，再讲原因。
- 能量化就量化，例如样本数量、命令名称、产物目录。
- 明确区分系统功能与研究实验。

这会让你的表达更像真实项目总结，而不是背模板。

## 七、最容易丢分的说法

下面这些话尽量不要说：

- “这个是模板自带的，我没怎么改。”
- “深度学习应该更好，只是我这次没有完全跑出来。”
- “NASA 那部分只是顺手加的。”
- “前端主要是为了好看。”

这些说法都会削弱你对项目的主导感，也会让老师怀疑项目完成度。

## 八、最后如何收口

如果答辩结束前需要一句总结，推荐这样说：

“这个项目最终完成的是一套可运行、可演示、可解释、可验证的毕业设计系统。它不仅具备行车日志业务管理和预测展示能力，也通过 NASA 电池寿命实验补充了研究深度，并通过测试、环境变量整理和一键启动脚本完成了交付层面的收口。”

这句话能够把工程实现、实验验证和交付完整性同时讲出来，比较适合作为答辩总结。
