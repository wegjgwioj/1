# 项目讲解稿设计

## 一、目标

本次文档整理的目标不是简单地把已有说明翻译成中文，而是重新组织一套更贴合毕业论文、答辩展示和项目学习的讲解稿。新的文档体系需要同时满足三个要求：

- 让老师或同学在不先读源码的前提下，也能迅速理解项目在解决什么问题。
- 让后续学习者能够按照清晰顺序进入代码，而不是面对大量目录无从下手。
- 让论文中的系统设计、系统实现、实验验证、部署收口四条线索与仓库中的真实文件形成对应关系。

## 二、设计原则

### 1. 以学习路径为主线

文档阅读顺序不再按代码目录机械拆分，而是按“先理解项目，再理解架构，再进入前后端与算法，再看部署与答辩”的学习路径组织。这样更符合第一次接触本项目的阅读习惯，也更适合在论文中作为系统说明材料使用。

### 2. 以思路和架构为主体

每份文档都优先回答三个问题：

- 这一部分为什么存在。
- 它在系统中承担什么职责。
- 阅读代码时应该先看哪些文件。

代码路径作为辅助证据出现，而不是把文档写成逐行注释式的“文件字典”。

### 3. 文件名全部中文且简单易懂

旧的英文命名文档全部删除，替换为中文名称。新文件名尽量使用“项目导读、系统架构讲解、后端实现讲解”这类能直接表达用途的名字，避免出现生硬缩写或中英混排。

### 4. 与论文表达保持一致

文档要明确区分以下几组边界，避免论文和答辩中出现概念混淆：

- 业务主系统与 NASA 电池寿命实验。
- 机器学习主链路与深度学习对比实验。
- 答辩版本的可运行交付与面向大规模数据的扩展性设计。

## 三、文档结构

本次计划整理如下中文文档：

- `docs/项目导读.md`
  说明项目定位、阅读顺序、代码目录入口和整体学习建议。
- `docs/系统架构讲解.md`
  说明系统六层架构、请求流、数据流、训练产物流和整体设计取舍。
- `docs/后端实现讲解.md`
  说明 Django 配置、接口组织、业务模型、管理命令和后端阅读顺序。
- `docs/前端实现讲解.md`
  说明 Vue 管理端页面结构、状态收口方式、前后端交互与学习顺序。
- `docs/预测与实验讲解.md`
  说明机器学习主模型、深度学习对比链路与 NASA SOH/RUL 实验。
- `docs/部署与运行讲解.md`
  说明 Windows 本地自举、环境变量、启动脚本和交付策略。
- `docs/测试与答辩讲解.md`
  说明测试覆盖、演示顺序、答辩高频问题和回答思路。
- `docs/论文写作讲解.md`
  说明论文各章如何对应系统实现，以及写作时的口径控制。
- `docs/相关文献整理.md`
  整理适合写入论文的深度学习相关文献，并说明和本项目的对应关系。

## 四、替换范围

以下旧文档将被删除：

- `docs/project-code-guide.md`
- `docs/system-architecture-and-thesis-guide.md`
- `docs/defense-interview-qa.md`
- `docs/thesis-related-work-papers.md`

旧文档中的有效内容会被吸收进新的中文讲解稿，但不会继续保留旧文件名。

## 五、预期效果

完成后，仓库中的文档体系会更接近一套“面向毕设交付的项目学习手册”。别人拿到项目后，可以先从导读理解总体目标，再顺着架构、后端、前端、预测实验、部署测试一路读下去，最后回到论文和答辩材料形成完整闭环。
