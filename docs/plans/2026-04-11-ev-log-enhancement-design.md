# 电动汽车行车日志系统功能补齐设计

**日期：** 2026-04-11

**背景：**
当前项目已经具备行车日志管理、统计分析看板和电池寿命预测能力，但与论文及 [1.txt](D:/GraduateProject/Project/1/docs/待提交/1.txt) 中描述相比，仍缺少收藏、评论反馈、基于用户偏好的推荐机制。同时，用户与配置相关代码中存在部分明显的空壳或错误实现，影响系统完整性与可交付性。

## 一、目标

本次补齐以“不推翻现有 Django + Vue 模板结构”为前提，完成以下目标：

1. 为 `drivinglog` 增加真实可用的收藏能力。
2. 为 `drivinglog` 增加真实可用的评论反馈能力，并兼容前端现有回复评论机制。
3. 将 `drivinglog/autoSort2` 补成“基于用户偏好的推荐结果”接口。
4. 修复若干明显半成品问题，包括：
   - `users_accountList` 返回未定义变量
   - `user/users importExcel` 空壳
   - 多处 `autoSort2` 空返回
5. 引入 Docker 化数据库启动方案，使项目在另一台电脑上更容易迁移和运行。

## 二、总体方案

### 1. 核心业务保持不变

保留现有 3 条主线：

1. 行车日志管理
2. 行车数据清洗与可视化分析
3. 电池寿命预测

在此基础上新增 3 条补齐能力：

1. 收藏
2. 评论反馈
3. 偏好推荐

### 2. 改造原则

1. 尽量复用现有自动注册路由和模板配置机制。
2. 优先兼容当前前端交互，不重写整页。
3. 推荐采用“可解释规则法”，而非新增复杂模型。
4. 数据库不依赖宿主机安装 MySQL，统一切到 Docker 容器中运行。

## 三、数据模型设计

### 1. 新增 `storeup`

用途：记录用户对 `drivinglog` 的收藏。

建议字段：

- `addtime`
- `userid`
- `refid`
- `tablename`
- `name`
- `picture`
- `type`
- `inteltype`

说明：

1. `tablename` 默认记录为 `drivinglog`，保留通用扩展能力。
2. `name` 用于前端列表展示，优先使用车辆编号或车辆编号与车型组合。
3. `picture` 当前项目暂无真实封面图，先兼容保留为空字符串。

### 2. 新增 `discussdrivinglog`

用途：记录用户对某条行车日志的评论与反馈。

建议字段：

- `addtime`
- `refid`
- `userid`
- `avatarurl`
- `nickname`
- `content`
- `reply`
- `isreply`
- `status`

说明：

1. `reply` 直接保存 JSON 字符串，兼容前端当前“回复评论”的实现。
2. `status` 用于管理员审核或显示状态扩展，当前可默认“正常”。

### 3. 扩展 `drivinglog`

新增两个缓存字段：

- `storeupnum`
- `discussnum`

用途：

1. 收藏/评论后同步更新数量。
2. 便于前端展示与简单统计。
3. 可作为推荐的弱辅助特征。

## 四、推荐机制设计

### 1. 目标

满足论文与进度说明中“基于用户偏好的推荐机制”这一描述，同时保证实现简单、稳定、可解释。

### 2. 偏好来源

用户偏好由以下行为组合生成：

1. 收藏过的 `vehiclemodel`
2. 收藏过的 `drivingroute`
3. 最近浏览过的日志对应的 `vehiclemodel`
4. 最近浏览过的日志对应的 `drivingroute`

### 3. 打分规则

候选数据：`drivinglog` 中未被当前用户收藏的记录。

打分方式：

1. 车型命中收藏偏好：`+3`
2. 路线命中收藏偏好：`+2`
3. 车型命中最近浏览偏好：`+1`
4. 路线命中最近浏览偏好：`+1`
5. 收藏数或评论数较高可作为弱权重：`+0~1`
6. 最新数据适当加权：`+1`

最终按“偏好分数降序 + 最新时间降序”返回。

### 4. 兜底逻辑

如果用户没有收藏/浏览行为，则退化为默认最新日志列表，保证接口始终稳定可用。

## 六、数据库与部署方案

### 1. 数据库部署要求

数据库必须运行在 Docker 容器中，避免依赖宿主机本地安装 MySQL。

### 2. 方案

新增：

1. `docker-compose.yml`
2. `.env.example`
3. `db/init/002_feature_patch.sql`

做法：

1. 使用 `mysql:8.0` 容器。
2. 使用仓库已有 `db/diandong5k56la1f.sql` 初始化基础库。
3. 使用新增补丁 SQL 创建功能新增表与字段。
4. 使用 Docker 命名卷保存数据库数据，避免绑死机器路径。

### 3. 配置读取

改造配置读取逻辑为：

1. 优先读取环境变量
2. 环境变量缺失时回退到 `config.ini`

主要变量：

- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`

这样在新电脑上只需：

1. 复制 `.env.example` 为 `.env`
2. 执行 `docker compose up -d db`
