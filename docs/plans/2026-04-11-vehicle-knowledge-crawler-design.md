# 车型知识采集功能设计

**日期：** 2026-04-11

**背景：**
当前项目已经具备行车日志管理、统计分析和寿命预测能力，但“爬虫”只停留在模板遗留接口层面，仓库内没有真实可运行的爬虫工程，也没有采集结果的持久化与展示页面。为了让“数据采集”链路在代码上真正成立，本次将爬虫功能改造成“车型知识采集与补全”。

## 一、目标

本次实现聚焦以下 4 个目标：

1. 在项目内落地真实可运行的车型知识采集逻辑。
2. 将采集结果持久化到数据库，而不是仅返回临时文本。
3. 为管理员和用户提供可查看的“车型知识库”页面。
4. 为行车日志提供“按车型触发采集”的入口，形成从日志到知识补全的闭环。

## 二、方案比较

### 方案 A：直接把爬到的内容写回 `drivinglog`

优点：

1. 开发量最小。
2. 不需要新增页面。

缺点：

1. 一种车型会在多条日志里重复存储，数据冗余大。
2. 日志表与知识表职责混杂，后续维护差。
3. 不方便批量采集和统一查看采集结果。

### 方案 B：新增独立 `vehicleknowledge` 表并提供采集入口

优点：

1. 车型知识与行车日志职责分离，结构清晰。
2. 可按车型去重并做增量更新。
3. 前端可以单独展示“车型知识库”，更适合答辩演示。
4. 后续如果替换数据源或补更多字段，改动面更小。

缺点：

1. 需要新增模型、接口、前端配置和菜单。
2. 实现量比方案 A 略大。

### 方案 C：引入完整 Scrapy 子工程

优点：

1. 形式上最接近传统“爬虫工程”。
2. 适合后续扩展多站点采集。

缺点：

1. 当前仓库没有 Scrapy 结构，改造成本高。
2. 额外依赖、运行方式和部署复杂度都会上升。
3. 对毕业设计当前阶段来说明显过重。

## 三、推荐方案

采用 **方案 B：独立 `vehicleknowledge` 表 + 项目内轻量采集器**。

原因：

1. 它能把“爬虫”真正做成可运行功能，而不是保留空接口。
2. 它足够轻量，不会破坏现有 Django + Vue 模板结构。
3. 它能解释为什么项目需要爬虫：不是采原始行车遥测，而是采公开车型知识，用来补全系统信息展示与辅助分析。

## 四、数据模型设计

新增表：`vehicleknowledge`

建议字段：

1. `vehiclemodel`：车型名称
2. `manufacturer`：厂商/品牌
3. `batterytype`：电池类型
4. `officialrange`：官方续航
5. `chargeinfo`：充电信息
6. `summary`：车型简介
7. `sourceurl`：采集来源页面
8. `crawlstatus`：采集状态，成功/失败
9. `crawltime`：最近采集时间
10. `rawdata`：原始结构化信息 JSON

设计要点：

1. `vehiclemodel` 作为业务主键使用，接口层做 upsert。
2. `rawdata` 保留原始信息，便于后续二次解析。
3. 采集失败也保留记录，方便前端看到失败状态和重试。

## 五、采集流程设计

### 1. 数据来源

首版采用 **百度百科车型词条页面** 作为公开数据源。

原因：

1. 当前项目已有百科抓取代码和 `requests + lxml` 依赖基础。
2. 公开页面更容易解释为“网络信息采集”。
3. 采集内容更适合作为车型知识补全。

### 2. 采集流程

1. 接收车型名，例如 `比亚迪海豹`、`Model Y`
2. 请求 `https://baike.baidu.com/item/<车型名>`
3. 解析简介区块和基础信息区块
4. 从基础信息中抽取厂商、电池类型、续航、充电时间等字段
5. 将结构化结果写入 `vehicleknowledge`
6. 返回采集状态与入库结果

### 3. 失败兜底

以下情况视为失败：

1. 页面请求失败
2. 页面被重定向到异常内容
3. 页面能打开但未解析出简介与基础信息

失败时：

1. 保存失败状态和来源地址
2. 返回明确错误信息
3. 前端允许再次采集

## 六、后端接口设计

除了 `vehicleknowledge` 的常规 CRUD 外，新增 3 个接口：

1. `POST /vehicleknowledge/crawlByModel`
   - 输入：`vehiclemodel`
   - 作用：按车型立即采集并写库

2. `POST /vehicleknowledge/crawlBatch`
   - 输入：可为空
   - 作用：扫描 `drivinglog` 中出现过的车型名，批量采集

3. `GET /vehicleknowledge/queryByModel`
   - 输入：`vehiclemodel`
   - 作用：按车型快速查询已有知识记录

## 七、前端交互设计

### 1. 新增菜单

新增 `车型知识库` 页面。

页面用途：

1. 查看已采集的车型知识
2. 手动新增或修改知识
3. 批量触发采集
4. 对失败记录进行重采

### 2. 行车日志页入口

在 `行车日志` 列表增加按钮：

1. `采集车型知识`

点击后：

1. 读取当前日志的 `vehiclemodel`
2. 调用 `crawlByModel`
3. 成功后提示已采集

### 3. 车型知识页入口

在 `车型知识库` 列表增加：

1. 表头按钮：`批量采集`
2. 行按钮：`重新采集`

## 八、测试设计

本次测试重点覆盖：

1. 车型知识模型存在且字段正确
2. HTML 解析逻辑能提取简介和基础信息
3. `crawlByModel` 能在 mock 网络响应下写入数据库
4. `crawlBatch` 能对 `drivinglog` 中不同车型进行去重采集
5. 前端构建通过，后端测试通过

## 九、暂不做的内容

本次明确不做：

1. 引入完整 Scrapy 子工程
2. 多站点聚合采集
3. 异步任务队列
4. 自动定时采集
5. 用爬虫直接生成行车日志原始遥测数据

## 十、预期结果

完成后，项目中的“爬虫”将不再是空壳接口，而是：

1. 可以真正发起网络采集
2. 有独立的数据表保存结果
3. 有前端入口触发和查看结果
4. 能合理解释其在系统中的用途
