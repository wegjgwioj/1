# 部署与运行讲解

## 一、为什么本项目强调 Windows 本地自举

很多毕业设计项目在提交给别人之后，最大的问题不是功能不够，而是别人 clone 下来根本跑不起来。本项目后期专门把部署方式整理成“Windows 本地自举优先”，原因主要有三点：

- 答辩现场最常见的环境就是 Windows 电脑。
- 很多老师和同学没有 Docker 环境，也不希望再额外装虚拟化工具。
- 本地一键启动更能体现项目交付完整度。

因此，这个项目不是把部署写成“理论上能运行”，而是尽量把“拿到仓库就能按步骤启动”做成文档和脚本的一部分。

## 二、部署相关文件应该先看哪些

关键文件主要集中在以下位置：

- `README.md`
- `bin/bootstrap_local.ps1`
- `bin/start_project.ps1`
- `bin/start_backend.ps1`
- `bin/bootstrap_local_db.py`
- `.env.example`
- `config.ini`

这些文件共同构成了本项目的本地交付方案。

## 三、首次启动的基本流程

第一次拿到项目后，推荐流程如下：

1. 安装 Python 3.11、Node.js 18+、MySQL、可选 Redis。
2. 执行 `bin/bootstrap_local.ps1`。
3. 让脚本自动创建虚拟环境、安装依赖、生成环境变量文件、初始化数据库并启动前后端。

这里的关键不是记住命令，而是理解“为什么要把这些步骤都收口到脚本里”。因为一旦把部署流程交给人工逐条手动执行，就会出现版本不一致、漏装依赖、端口混乱、环境变量不统一等问题。

## 四、环境变量为什么被专门统一

项目中环境变量的整理是一次很重要的工程化收口，主要解决以下问题：

- 后端和前端分别用不同方式读配置，容易错位。
- 换电脑后，数据库地址和端口经常忘记改。
- 前端接口地址硬编码，改一次端口要改多个地方。

现在的做法是：

- 后端优先读取 `.env`，不足时回退到 `config.ini`。
- 前端通过 `VITE_API_BASE_URL` 等变量收口接口地址。
- `README.md` 对常用变量做统一说明。

这种设计让“环境一致性”从口头提醒变成了可落地的项目约定。

## 五、为什么仍然保留 Docker 文件

仓库里还保留了 `docker-compose.yml` 和 Docker 相关文件，但它们不再是主推荐方式。这一点在答辩时也可以讲得很自然：

“项目在整理阶段保留了容器化方案作为补充路径，但考虑到答辩和本地演示的普遍环境，当前交付版本优先采用 Windows 本地自举。”

这样既说明你考虑过多种部署方式，又不会让老师误以为项目必须依赖 Docker 才能跑。

## 六、部署这一部分在学习和论文里的意义

对学习者来说，这部分说明了一个成熟项目不只是有代码和页面，还需要有清晰的运行入口。对论文来说，这部分则说明项目已经完成了从开发态到交付态的收口，不是停留在“本机作者环境可运行”的阶段。

如果要写进论文，可以这样表达：

“为提高系统的可迁移性与答辩演示稳定性，本文在完成核心功能开发后，对项目的本地部署流程进行了统一整理，形成以 Windows 本地自举为主的一键启动方案，并通过环境变量收口与脚本封装降低了系统的部署复杂度。”

这类表述会让老师看到你不仅做了功能，也做了工程交付。
