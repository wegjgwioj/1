# 前端实现讲解

## 一、前端的任务不是“把接口显示出来”

在这个项目里，前端承担的任务比普通管理系统更重一些。它不仅要把数据展示出来，还要把业务管理、预测分析、实验展示和演示路线组织成一个顺畅的交互流程。因此学习前端时，重点要看三件事：

- 页面是如何按业务主线组织的。
- 状态和会话是如何被统一管理的。
- 前端如何把训练结果和实验结果变成适合答辩展示的界面。

## 二、前端的入口结构

前端位于 `templates/front/admin/`，核心入口包括：

- `src/main.js`
- `src/App.vue`
- `src/router/`
- `src/utils/`
- `src/views/`

学习时不要一开始就看某个页面组件，而应该先从应用入口、路由和公共工具入手。这样才能理解页面之间如何跳转、请求如何发送、登录状态如何维持。

## 三、页面结构如何围绕系统主线展开

### 1. 首页与总览

首页和总览页面的作用不是单纯“好看”，而是帮助老师和用户在进入系统后立刻知道这个项目有哪些核心能力。因此首页对应的是“项目全景入口”，适合在答辩开场阶段展示。

### 2. 行车日志列表页

列表页是业务主线最强的页面之一，它体现了系统并不是只会做预测，而是先有数据管理能力，再有分析能力。这里通常会承载：

- 查询与筛选
- 详情查看
- 收藏与评论
- 车型知识采集

这部分最适合说明系统的业务完整性。

### 3. 预测工作台

预测工作台是整个前端中最能体现“智能分析”价值的页面。它把表单输入、模型输出、场景预测和对比展示整合在一起，是答辩演示时最核心的页面之一。

### 4. NASA 实验页

NASA 页面承担的职责不是日常业务操作，而是向老师说明项目具有研究实验能力。它通常会展示：

- 数据准备结果
- SOH / RUL 训练指标
- 图像结果
- 实验说明文字

因此这一页更像“研究补强展示页”，而不是普通业务页面。

## 四、前端公共工具为什么值得重点看

### 1. `src/utils/http.js`

这是前后端交互的统一入口。只要接口调用逻辑从这里收口，后续无论是统一错误处理、统一鉴权头、统一基础地址配置，都会更容易维护。

### 2. `src/utils/base.js`

它负责前端对后端基础地址的读取与拼接，是环境变量收口的重要位置。学习这个文件时要理解一点：本项目后期专门做过环境变量统一，就是为了避免换电脑、换端口、换部署方式时前端失联。

### 3. `src/utils/adminSession.js`

这是本项目一个很有代表性的收口点。之前系统存在旧标签页残留、退出登录不彻底、鉴权异常提示混乱等问题，后来通过会话清理工具统一收口。这个文件很适合在讲前端稳定性时重点提一下，因为它体现的是“工程化修整能力”。

## 五、前端为什么看起来更适合答辩了

这不是一句审美判断，而是一次系统化整理的结果。前端被重新整理后，重点改进体现在：

- 页面结构更清楚，老师更容易跟着演示节奏走。
- 模块之间的视觉关系更统一，不会出现信息挤成一团的感觉。
- 预测工作台和 NASA 页面更突出核心内容，而不是把所有元素平铺在一起。
- 登录态清理更稳，减少旧调试状态残留导致的异常。

换句话说，这次前端整理的目标不是“做成炫技页面”，而是“做成一套答辩时不容易出事故的展示界面”。

## 六、前端阅读顺序建议

建议按下面顺序进入源码：

1. `src/main.js`
2. `src/router/`
3. `src/utils/http.js`
4. `src/utils/base.js`
5. `src/utils/adminSession.js`
6. `src/views/home/`
7. `src/views/list/`
8. `src/views/forecast-workbench/`
9. `src/views/board/`

## 七、前端在论文里怎么写更自然

论文中不建议把前端写成“基于 Vue 实现若干页面”。更好的写法是：

“前端采用 Vue 3 构建管理端界面，通过统一路由、统一请求工具和统一会话管理方式，将行车日志管理、预测工作台、车型知识采集和 NASA 实验结果展示整合为一套面向演示与交互的可视化系统。前端设计在保证功能完整性的基础上，进一步强调页面结构清晰、核心信息突出与答辩演示稳定性。”

这样就能把“前端优化”和“答辩交付”自然联系起来，而不是只停留在页面数量的罗列上。
