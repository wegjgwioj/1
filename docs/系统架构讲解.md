# 系统架构讲解

## 一、为什么本项目要按分层架构理解

这个项目包含后台业务、可视化页面、模型训练、实验管理和本地部署脚本。如果只按目录去看，很容易觉得内容分散。更好的理解方式是把它看成一套六层架构：

- 数据来源层
- 数据治理层
- 数据存储层
- 算法分析层
- 服务接口层
- 前端展示层

这样的好处是，无论你是在写论文、准备答辩，还是带别人学习项目，都能沿着同一条逻辑解释系统，而不是在“页面、脚本、模型、文档”之间来回跳。

## 二、六层架构分别负责什么

### 1. 数据来源层

项目的数据来源有三类：

- 业务演示数据，主要来自仓库中的 Excel 样例与数据库初始化数据。
- 公开车型资料，主要通过车型知识采集功能补充到 `vehicleknowledge` 表。
- NASA 公开电池老化数据，用于 SOH 和 RUL 实验。

这一层回答的是“数据从哪里来”的问题。

### 2. 数据治理层

原始数据并不能直接用于业务展示和模型训练，所以项目在这一层做了数据清洗、字段标准化、结构转换和序列构造。对应的关键位置包括：

- `main/excel_sources.py`
- `main/ml/feature_engineering.py`
- `main/dl/sequence_dataset.py`
- `main/battery_life/nasa_dataset.py`
- `main/management/commands/prepare_demo_data.py`
- `main/management/commands/prepare_sequence_data.py`

这一层回答的是“数据如何从原始状态变成可用状态”的问题。

### 3. 数据存储层

项目的存储不是单一数据库，而是由三部分共同组成：

- MySQL 负责结构化业务数据。
- `artifacts/` 负责保存模型文件、指标报告和实验图表。
- Redis 作为可选缓存存在，不可用时系统会自动回退到本地缓存。

这里要特别注意，训练产物放在 `artifacts/` 是一个非常重要的设计取舍。因为答辩和系统展示需要稳定、可复用的结果，而不是每次启动都重新训练。

### 4. 算法分析层

算法分析层包含三条链路：

- `main/ml/`：机器学习主预测链路。
- `main/dl/`：深度学习对比实验链路。
- `main/battery_life/`：NASA 电池寿命实验链路。

它们共同服务于“系统有分析能力”这一目标，但承担的角色不同。机器学习主链路是主展示输出，深度学习链路负责对比实验，NASA 链路负责研究补强。

### 5. 服务接口层

服务接口层由 Django 负责，关键组成包括：

- `dj2/settings.py`：全局配置与组件装配。
- `main/urls.py`：接口注册与路由收口。
- `xmiddleware/xparam.py`、`xmiddleware/xauth.py`：请求参数与鉴权逻辑。
- `main/*_v.py`：具体业务接口实现。
- `main/management/commands/`：训练、评估、数据准备、环境整理命令。

这一层决定了前端如何访问数据、模型如何被管理命令调度、页面操作如何落到后端逻辑上。

### 6. 前端展示层

前端管理端位于 `templates/front/admin/`，承担的是“把后端能力组织成可演示界面”的任务。这里不只是页面美化，更重要的是把日志管理、预测工作台、NASA 实验页、知识采集页等能力合理串起来。

## 三、一次完整请求是怎么跑通的

以预测工作台为例，一次请求大致经历下面过程：

1. 用户在前端页面输入车辆参数与场景信息。
2. 前端通过 `src/utils/http.js` 发送请求，并附带登录态。
3. 中间件在后端解析 `token`，识别当前用户身份。
4. 请求进入 `main/Drivinglogforecast_v.py` 对应接口。
5. 接口从 `artifacts/` 中加载模型和指标，必要时调用预测器生成结果。
6. 后端把预测值、风险等级、主要影响因素等信息返回前端。
7. 前端再把结果组织成卡片、图表和说明文字展示给用户。

这条链路说明，本项目并不是“算法代码单独放一边”，而是把训练产物真正接入了系统服务流程。

## 四、训练产物为什么要单独保存

很多课程设计项目在答辩时会遇到一个问题：演示现场重新训练模型耗时太长，甚至因为环境差异直接失败。本项目通过 `artifacts/` 目录解决了这个问题：

- 模型文件可以提前训练好并保留。
- 指标文件可以直接供页面展示。
- NASA 实验图像和报告也能提前生成。

这样做的意义不只是“方便”，而是把“实验结果可复现”变成了系统交付的一部分。

## 五、系统的核心设计取舍

### 1. 先保证可运行，再追求扩展性

论文中可以讨论大规模数据与分布式分析，但当前交付版本优先保证 Windows 本地环境可启动、可演示、可复现实验。这种取舍非常适合毕业设计，因为老师最先看的不是远期扩展能力，而是你现在能不能把系统完整跑起来。

### 2. 先保证主链路稳定，再增加实验深度

业务预测采用机器学习主链路，是为了稳定和可解释。深度学习与 NASA 实验作为补充，不是因为它们不重要，而是因为系统需要先有一条可靠主线，再去扩展研究深度。

### 3. 先保证理解顺畅，再补代码细节

学习这个项目时也应该遵循同样原则。先把架构和数据流看懂，再去进入某一个模型文件或页面组件，会更容易理解为什么代码要写成现在这样。

## 六、论文和答辩里怎么讲这一套架构

如果你要把这一节写进论文或答辩，建议统一口径为：

“本系统采用分层架构设计，从数据来源、数据治理、数据存储、算法分析、服务接口和前端展示六个层面构建完整的电动汽车行车日志分析平台。系统既支持业务层面的日志管理与预测展示，也支持基于公开数据集的电池 SOH/RUL 研究实验，从而实现工程应用与研究验证的结合。”

这句话基本可以覆盖整个系统架构图的口径，也方便后续自然过渡到后端实现、前端实现和实验设计章节。
